分析日期：2026-05-13

## 1. 项目概况

当前项目是一个面向景区预约、导览和后台管理的智慧文旅系统，整体采用三端架构：

- 后端：`xfs-backend`，Spring Boot 2.7.18 + MyBatis-Plus + Redis + Druid + JWT + Hutool。
- 管理后台：`xfs-admin`，Vue 3 + Vite + Element Plus + ECharts + Axios。
- 游客端：`xfs-miniapp`，UniApp + Vue 3。

项目已经具备毕业设计演示所需的核心能力，包括景区展示、游客预约、二维码核销、后台统计大屏、AI 导览问答、管理员登录鉴权、游客手机号验证码登录等。

## 3. 主要问题与风险

### 3.1 验证码接口直接返回验证码

位置：`xfs-backend/src/main/java/com/xfs/xfsbackend/controller/TouristAuthController.java`

当前 `/api/tourist/sendCode` 会将验证码直接拼接在接口响应里：

```java
return Result.success("验证码已发送：" + code);
```

这适合毕业设计演示，但不适合真实上线。真实系统中验证码应通过短信服务发送，接口只返回“验证码已发送”，不能把验证码暴露给前端。

建议：

- 演示环境保留模拟验证码，但通过配置开关控制。
- 生产环境接入短信服务。
- 增加手机号发送频率限制，例如同一手机号 60 秒内只能发送一次。

### 3.2 预约重复校验存在并发竞态

位置：`xfs-backend/src/main/java/com/xfs/xfsbackend/service/ReserveRecordService.java`

当前预约流程大致为：

1. 查询是否已有重复预约。
2. Redis 扣减库存。
3. 写入预约订单。

如果同一用户并发提交两次预约请求，两个请求可能同时通过重复预约查询，然后分别扣库存并写入订单。

建议：

- 数据库增加唯一约束：`tourist_id + area_id + slot_id + reserve_date + status`。
- 服务层捕获唯一键冲突，返回“请勿重复预约”。
- 如果写库失败，需要继续确保 Redis 库存被正确回滚。

### 3.3 Redis 库存 key 缺少景区维度

位置：`xfs-backend/src/main/java/com/xfs/xfsbackend/service/ReserveRecordService.java`

当前库存 key 形如：

```text
ticket_stock:{date}:{slotId}
```

这意味着不同景区在同一天同一时段会共享同一个库存池。例如 A 景区 08:00-10:00 的预约可能消耗 B 景区同一时段的名额。

建议改为：

```text
ticket_stock:{areaId}:{date}:{slotId}
```

同时，库存预热接口也应按景区、日期、时段维度初始化库存。

### 3.4 Redis 库存 key 没有过期时间

当前库存初始化使用 `setIfAbsent`，但没有设置 TTL。随着日期增加，Redis 中会积累大量过期日期的库存 key。

建议：

- 对库存 key 设置过期时间，例如游玩日期结束后 1 到 7 天自动过期。
- 增加定时任务清理历史库存 key。

### 3.5 生产配置仍存在默认弱口令和默认密钥

位置：`xfs-backend/src/main/resources/application.yml`

当前配置中存在：

- 数据库默认密码：`${DB_PASSWORD:20201211}`
- Druid 默认账号密码：`admin/admin123`
- JWT 默认密钥：`${JWT_SECRET:XFS_SECRET_2026_GRADUATION_PROJECT_DEFAULT}`
- AI Key 默认占位：`${AI_API_KEY:your-real-key}`

这些配置便于本地运行，但不能直接用于生产环境。

建议：

- 增加 `application-prod.yml`。
- 生产环境强制从环境变量读取敏感配置。
- 如果生产环境缺少 `DB_PASSWORD`、`JWT_SECRET`、`AI_API_KEY`，应用应启动失败。
- Druid 监控页在生产环境关闭，或至少配置强密码和访问白名单。

### 3.6 日志可能记录敏感参数

位置：`xfs-backend/src/main/java/com/xfs/xfsbackend/common/LogAspect.java`

当前日志切面会记录控制器入参：

```java
params = Arrays.toString(joinPoint.getArgs());
```

这可能把密码、手机号、验证码、Token、AI 提问内容等写入日志。

建议：

- 对 `password`、`code`、`phone`、`Authorization` 等字段做脱敏。
- 日志重点记录接口、用户 id、角色、耗时、状态码、traceId。
- AI 对话内容建议单独做合规处理，不直接进入普通业务日志。

### 3.7 小程序登录逻辑仍有重复残留

位置：`xfs-miniapp/pages/index/index.vue`

项目已经抽取了公共登录逻辑：

```text
xfs-miniapp/utils/auth.js
```

但首页 `goToProfile` 仍保留了 `promptLoginDirect()` 的整套登录流程。后续如果验证码、登录态、错误处理策略变化，很容易出现两套逻辑不一致。

建议：

- 首页“个人中心”和“我的预约”统一使用 `ensureLogin()`。
- 删除 `promptLoginDirect()`。
- 所有游客端登录入口统一收敛到 `utils/auth.js`。

### 3.8 小程序生产 API 地址仍是占位域名

位置：`xfs-miniapp/api/request.js`

当前生产环境地址为：

```js
https://your-server.com
```

如果直接打包上线，游客端会请求错误域名。

建议：

- 使用构建环境变量注入后端地址。
- 提供 `.env.example` 说明开发、测试、生产地址。
- 在部署文档中明确小程序合法域名配置。

### 3.9 业务错误统一返回 `code=500`

位置：`xfs-backend/src/main/java/com/xfs/xfsbackend/common/Result.java`

当前 `Result.error()` 固定返回：

```java
code = 500
```

这会导致参数错误、未登录、权限不足、业务冲突等都被包装成 500，前端难以做精确处理。

建议引入更细的业务码：

- `400`：参数错误。
- `401`：未登录或 token 失效。
- `403`：权限不足。
- `409`：重复预约、库存冲突。
- `500`：真正的服务器异常。

## 5. 游客端分析

游客端当前具备：

- 景区列表。
- 热门推荐。
- 景区详情。
- 子景点展示。
- 门票预约。
- 我的预约。
- 个人中心。
- AI 助手。

建议完善：

1. 预约页展示真实余票，而不是固定显示“余量充足”。
2. 增加取消预约能力，并将库存释放回 Redis。
3. 增加预约详情页，展示景区、日期、时段、二维码、订单状态。
4. 处理二维码过期、订单已核销、订单已取消等状态。
5. 增加游客手机号换绑或退出登录能力。

## 6. AI 导览能力建议

当前 AI 能力已经接入真实大模型接口，支持多轮对话和系统 Prompt。

建议进一步增强：

1. 使用本地景区文档构建知识库，让 AI 回答基于项目已有景区资料。
2. 增加 RAG 检索流程，减少模型编造。
3. 给回答附带来源，例如“根据穿岩山景区资料”。
4. 对用户问题做旅游领域识别，过滤明显无关或违规问题。
5. 为管理端提供 AI 问答日志分析，帮助优化景区服务。

## 7. 数据与统计能力建议

当前统计包括今日预约、累计核销、景区数量、核销率、景区热度、最近 7 天趋势。

建议增加：

1. 日期范围筛选。
2. 按景区、时段、预约状态过滤。
3. 预约量、核销量、取消量、过期量趋势。
4. 库存预警，例如某景区某时段余票低于阈值。
5. Excel 导出能力，便于毕业答辩展示和实际运营使用。

## 8. 安全与部署建议

优先级较高的安全和部署事项：

1. 生产环境禁用默认 JWT 密钥。
2. 生产环境禁用 Druid 默认账号密码。
3. 管理员登录增加失败次数限制。
4. 游客验证码发送增加手机号和 IP 限流。
5. CORS 白名单从配置读取，不要长期只允许 localhost。
6. 补充 Redis、MySQL、后端、管理端、小程序的启动顺序文档。
7. 提供 `.env.example` 和部署检查清单。

- 验证码生产环境不再直接返回给前端。

- 核销记录和操作日志。

### P2：体验与专业度提升

- AI 知识库/RAG。
- 统计筛选与导出。
- 图片上传。
- ECharts 按需加载和构建体积优化。

## 10. 结论

当前项目已经具备完整毕业设计演示基础，核心链路包括“游客浏览景区 -> 登录 -> 预约 -> 生成二维码 -> 后台核销 -> 统计展示 -> AI 导览”。从演示角度看，项目完成度较高。

如果按准生产系统标准评估，当前最需要补强的是库存一致性、并发重复预约控制、生产配置安全、验证码安全、操作审计和预约订单管理。建议先处理 P0 项，再继续完善后台运营能力和 AI 知识库能力。
